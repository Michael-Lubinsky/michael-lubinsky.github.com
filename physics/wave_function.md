## Schrödinger Solver

The wavefunction ψ(x,t) is a **spatial field** — you can plot it, 
animate its time evolution inside a potential well, show |ψ|² as a probability density in real-time.  
Users can literally draw a potential V(x) with their mouse and watch the eigenstates appear.  
That maps naturally to an interactive UI.  

 
- **Schrödinger picture**: states |ψ(t)⟩ evolve, operators Ô are static
- **Heisenberg picture**: states are frozen, operators Ô(t) carry all time dependence, governed by the equation of motion: dÔ/dt = (i/ℏ)[H, Ô]

The choice is purely one of computational/conceptual convenience for a given problem.

## Where Heisenberg Matrix Mechanics Is Actually Superior

| Problem type | Better formulation | Why |
|---|---|---|
| Spin systems, qubits | Heisenberg | Hilbert space is **finite-dimensional** — matrices are exact, not discretized |
| Finite lattice models | Heisenberg | Tight-binding Hamiltonian *is* a matrix — exact diagonalization is natural |
| Quantum optics (Fock space) | Heisenberg | Ladder operators â, â† are matrices in truncated Fock space |
| Harmonic oscillator | Heisenberg | Analytic operator algebra is cleaner than solving the PDE |
| Continuous spatial potentials | Schrödinger | PDE on a grid, spatial intuition, easy to visualize |

Notice that my **tight-binding model explorer** idea (#2 in the list) is *already* Heisenberg matrix mechanics — you build H as a matrix and diagonalize it. I just didn't frame it that way.

## A More Honest Project Framing

A genuinely interesting project would be a **unified visualizer** that shows *both pictures simultaneously*:

- Left panel: wavefunction ψ(x,t) evolving in Schrödinger picture
- Right panel: the same physics shown as matrix elements ⟨m|Ô(t)|n⟩ evolving in Heisenberg picture
- Show explicitly how ⟨x(t)⟩ computed via Ehrenfest's theorem matches between both

This would be **more original** than yet another Schrödinger solver, 
and would address a real pedagogical gap — most students never viscerally see 
that the two pictures are the same physics. 
The matrix picture also connects directly to quantum computing 
(density matrices, Lindblad operators), making such a tool useful beyond just introductory QM.

The Schrödinger recommendation was the path of least resistance visually, not the most physically interesting or novel choice.

## 5. Solver approach: finite difference vs spectral

This is the most important design choice.

## My recommendation For MVP:

> Use **finite difference** as the primary solver.

Then later add:

> **split-step spectral** for special time-dependent cases.

Why:

* finite difference is easier to explain
* works well with arbitrary 1D potentials
* handles stationary and time-dependent modes with one Hamiltonian construction model
* easier for custom boundaries and future 2D extension

---

## 6. Finite difference approach

For 1D Schrödinger equation:

```text
H = -(ħ² / 2m) d²/dx² + V(x)
```

Discretize space:

* grid `x_0 ... x_{N-1}`
* second derivative via tridiagonal stencil

Then:

* kinetic term becomes sparse tridiagonal matrix
* potential term becomes diagonal matrix
* full Hamiltonian is sparse Hermitian

This is perfect for sparse linear algebra.

SciPy’s `eigsh` is specifically for real symmetric / complex Hermitian sparse eigenproblems, and ARPACK is designed to find a few eigenpairs of large sparse operators efficiently. ([SciPy Documentation][3])

## Stationary solver with finite difference

Use:

* sparse Hamiltonian
* `eigsh(H, k=num_states, which="SM")` or shift-invert near target energy

Output:

* eigenvalues
* eigenvectors
* normalization checks

## Time-dependent solver with finite difference

Use one of these:

### Option 1: Crank–Nicolson

Best MVP choice.

Pros:

* stable
* norm-preserving in practice for Hermitian systems
* standard and reliable

Update:

```text
(I + iΔtH/2ħ) ψ_{n+1} = (I - iΔtH/2ħ) ψ_n
```

Because `H` is sparse, each step is solving a sparse linear system.

### Option 2: Matrix exponential / Krylov

Good later for performance and elegance.

### Option 3: Explicit RK

Not recommended as primary method for Schrödinger evolution.

Pros of finite difference:

* arbitrary `V(x)`
* simple sparse matrix structure
* one code path for eigen + evolution
* extendable to 2D

Cons:

* needs finer grids for high accuracy
* can be slower than spectral for smooth periodic cases

## 7. Spectral approach

There are two distinct “spectral” ideas people mix together:

### A. basis expansion in eigenfunctions

Expand wavefunction in some known basis.

Good when:

* harmonic oscillator
* infinite well
* analytically convenient systems

Bad for a general interactive solver:

* basis depends on problem
* harder for arbitrary user potentials

### B. split-step Fourier method

Excellent for time evolution when Hamiltonian separates nicely into kinetic + potential parts.

Typical update:

```text
ψ(t+Δt) ≈ e^(-iVΔt/2ħ) F^-1 e^(-iTΔt/ħ) F e^(-iVΔt/2ħ) ψ(t)
```

Pros:

* very fast for smooth problems
* elegant
* highly accurate for many wave-packet simulations

Cons:

* most natural with periodic boundary assumptions
* less intuitive for arbitrary boundary conditions
* not ideal as the only method for eigenstate solving

## Recommendation

Use spectral only as an optional “fast evolution engine” later:

* for free particle
* harmonic oscillator
* periodic / large smooth domains

## Is it good idea to implement schrodinger equation solution for hydrogen atom?

⏺ It's a natural next step scientifically, but it's a much larger jump than it appears. Here's an
  honest assessment:

  Why it's appealing

  - Hydrogen is the canonical quantum mechanics problem
  - Exact analytic solution exists (Bohr energies Eₙ = -13.6 eV / n²) — great for validation
  - Would make the tool genuinely useful for chemistry/atomic physics courses
  - Strong JOSS paper story: "from 1D toy problems to real atoms"

  Why it's hard

  It's fundamentally a 3D problem.
  The full hydrogen atom lives in 3D. The standard simplification exploits spherical symmetry to
  reduce it to a 1D radial equation for each (n, l) pair:

  -½ d²u/dr² + [l(l+1)/2r² - 1/r] u = E·u

  where u(r) = r·R(r). This is solvable as a 1D problem — but it's a different 1D problem with a
  singular potential at r=0 and a semi-infinite domain [0, ∞).

  The current infrastructure doesn't handle this without changes:
  - Grid is uniform on [x_min, x_max] — needs r ∈ [0, ∞), truncated at some r_max
  - Boundary condition at r=0 is u(0)=0 (already Dirichlet, OK), but the 1/r singularity at the
  origin causes numerical problems with uniform grids
  - The 1/r² centrifugal term diverges at r=0 for l > 0
  - Need to solve for multiple (n,l) pairs and reconstruct the full orbital

  Visualization is a new problem entirely.
  - Radial wavefunctions R(r) are 1D and easy to plot
  - But users expect to see hydrogen orbitals — the 3D shapes (s, p, d orbitals)
  - Rendering |ψ(r,θ,φ)|² in 3D requires volume rendering or isosurface plots — a completely
  different visualization stack than what you have

  CLAUDE.md explicitly defers 2D (let alone 3D).
  The 3D orbital visualization would be a substantial new frontend capability.

  What's actually feasible now

  A radial hydrogen solver — solve the 1D radial equation for given (n, l), plot u(r) and R(r),
  compare energies to Eₙ = -1/(2n²) in atomic units. This is:

  - A new potential preset: V_eff(r) = -1/r + l(l+1)/(2r²)
  - A new grid type: semi-infinite [r_min, r_max] with small r_min (e.g. 0.01) to avoid the
  singularity
  - Validation against exact Bohr energies

  This is realistic without touching the 3D visualization problem at all.

## The Core Principle

Claude Code works best when it has **maximal context upfront and minimal ambiguity about decisions already made**. The enemy is Claude making architectural choices you didn't intend, which then get baked into 500 lines of code you have to undo.

⏺ Honest evaluation of each against the existing architecture (1D single-particle
  Schrödinger equation, spatial grid, Crank-Nicolson):

  ---
  Spin in Quantum Mechanics — Medium effort, but requires restructuring

  Spin is an internal discrete degree of freedom. The wavefunction becomes a 2-component
  spinor (ψ↑, ψ↓). Adding it to the existing 1D solver means the Hamiltonian becomes a 2×2
  block matrix — the current tridiagonal structure breaks. A pure spin-½ system (no spatial
   grid, just 2×2 matrices) is a completely separate simple module. Spin-orbit coupling in
  1D is medium-hard. Worth considering as a new standalone "two-level system" mode.

  Bloch Sphere — Medium effort, completely independent

  The Bloch sphere visualizes a two-level system |ψ⟩ = α|0⟩ + β|1⟩ as a point on a unit
  sphere. Time evolution under H = h·σ is a rotation. This has zero connection to the grid
  solver — it's a new mode with 2×2 matrix algebra and a 3D Plotly sphere visualization.
  Plotly supports 3D surface plots, so the frontend is feasible. This is the most
  self-contained feature on the list and highly educational for QM courses (Rabi
  oscillations, qubit gates).

  Berry Phase — Hard, subtle

  Geometric phase γ = i∮⟨n(R)|∇ᴿ|n(R)⟩·dR requires adiabatically varying a parameter around
   a closed loop and tracking the wavefunction phase. The phase is gauge-dependent and hard
   to make visually intuitive. A 1D demonstration (Zak phase in SSH model) requires
  periodic boundary conditions and band structure — significant departure from the current
  Dirichlet BC setup. Not recommended unless the project pivots toward topology.

  Josephson Effect — Out of scope

  Not a single-particle Schrödinger equation at all. It's a phenomenological ODE for the
  macroscopic phase difference between two superconductors: dφ/dt = 2eV/ħ, J = J_c sin(φ).
  Could be implemented as a standalone ODE solver, but it shares nothing with the existing
  code and would be misleading alongside a wavefunction solver. Wrong tool.

  Quantum Hall Effect — Out of scope (requires 2D)

  Requires a 2D spatial grid, a vector potential A in the Hamiltonian (Landau gauge), and
  Chern number computation to show the topology. CLAUDE.md explicitly defers 2D. Even 1D
  Landau levels require periodic boundary conditions. This is a multi-sprint project on its
   own.

  Magnetism — Out of scope (many-body)

  The Heisenberg/Ising models are many-body lattice spin models — completely different
  physics from the single-particle Schrödinger equation. For small chains (N ≤ 20 spins)
  exact diagonalization is feasible but shares nothing with the existing solver. Wrong
  abstraction level for this project.

## Feature-by-Feature Review

### ✅ Matrix formulation of QM — Strong Yes

**Fit with existing architecture: excellent.**

Your Hamiltonian builder already constructs H as a sparse matrix. The matrix formulation is not a new solver — it is a new **visualization layer** on top of what you already have.

You can show:

### ❌ Dirac relativistic equation — No for now

**Fit with existing architecture: poor.**

The Dirac equation requires:
- 4-component spinor wavefunctions (not scalar ψ)
- Coupling between particle and antiparticle components
- Relativistic dispersion — fundamentally different Hamiltonian structure
- New numerical methods (the naive finite difference approach has the **fermion doubling problem**)

This is essentially a separate application. The physics is beautiful but the implementation complexity is out of proportion with what you would gain. **Defer indefinitely.**

### ✅ Spin examples — Yes, high value

**Fit with existing architecture: good, with extension.**

Spin is pure matrix mechanics — finite-dimensional Hilbert space, no spatial grid needed. A spin-½ system lives in ℂ², a spin-1 in ℂ³. The Hamiltonian is a small matrix involving Pauli matrices:

$$\hat{H} = -\gamma B_z \hat{S}_z = -\frac{\gamma B_z \hbar}{2}\begin{pmatrix} 1 & 0 \\ 0 & -1 \end{pmatrix}$$

You can show:
- Spin precession on a Bloch sphere (animated)
- Rabi oscillations between |↑⟩ and |↓⟩ under oscillating field
- Stern-Gerlach measurement visualization
- Two-spin entanglement (Bell states)

This is a **self-contained module** — it does not need the spatial grid at all. Small scope, high educational payoff, directly connects to quantum computing. **Implement this.**

---

### ✅ Zeeman effect — Yes, natural extension

**Fit with existing architecture: excellent.**

The Zeeman effect is what happens to energy levels when you add a magnetic field term to an existing Hamiltonian:

$$\hat{H} = \hat{H}_0 + \mu_B B \hat{L}_z$$

In 1D this is limited, but it is a natural motivator for the **hydrogen atom** module (see below). The normal Zeeman effect (splitting of spectral lines) can be shown by:

1. Solving H₀ eigenstates (already implemented)
2. Adding the B field perturbation term to H
3. Showing how degenerate levels split as B increases — a slider for B field strength

This reuses your entire existing eigenvalue solver with minimal change. **Implement this alongside hydrogen atom.**

---

### ⚠️ Hydrogen atom wavefunction — Yes, but requires 2D/3D

**Fit with existing architecture: requires extension.**

The hydrogen atom in full 3D requires spherical coordinates and radial + angular parts. However the **radial equation** after separating angular momentum is a 1D problem:

$$\left[-\frac{1}{2}\frac{d^2}{dr^2} + \frac{l(l+1)}{2r^2} - \frac{1}{r}\right]u(r) = E\,u(r)$$

where u(r) = r·R(r). This is **directly solvable with your existing 1D solver** on a radial grid, for each angular momentum quantum number l. You get the exact hydrogen energy levels Eₙ = -1/2n² and the radial wavefunctions.

The angular parts (spherical harmonics Yₗᵐ) can be shown analytically as separate 3D visualizations.

This is a high-value feature with a clear path. **Implement the radial solver first, add 3D visualization later.**

---

### ⚠️ Ising model — Maybe, different physics

**Fit with existing architecture: partial.**

## Dirichlet boundary conditions

means the wavefunction is forced to be **exactly zero at the walls** of your simulation domain:

$$\psi(x_{\min}) = 0 \qquad \psi(x_{\max}) = 0$$

That is the entire definition. "Dirichlet" is just the mathematician's name for "fix the value at the boundary."

## Why your solver uses them

Your finite difference grid has a finite domain, say x ∈ [−10, 10]. The wavefunction has to have some value at x = −10 and x = 10. You have to choose something. Dirichlet (set to zero) is the simplest physically motivated choice — it is equivalent to putting **infinitely hard walls** at the edges of the domain.

This means every potential you simulate is secretly inside an infinite square well of width 20 a.u. For most potentials with bound states that decay to zero well before the walls, this has negligible effect. For scattering potentials (step, gaussian barrier) it is the source of the fake discrete energy levels we discussed earlier.

---

## The three common choices

| Boundary condition | Mathematical meaning | Physical meaning |
|---|---|---|
| **Dirichlet** | ψ = 0 at walls | Infinite potential walls — particle cannot escape |
| **Periodic** | ψ(x_min) = ψ(x_max) | Particle on a ring — needed for Kronig-Penney / band theory |
| **Absorbing** | ψ → 0 smoothly near walls | Particle escapes — needed for true scattering problems |

Your current solver uses Dirichlet. Periodic BCs are what you would need to add for the Kronig-Penney band theory feature. Absorbing BCs are the most complex and would be needed to correctly handle the step potential and Gaussian barrier as scattering problems.

## The Physics

A qubit is a two-level quantum system. The state is:

$$|\psi\rangle = \alpha|0\rangle + \beta|1\rangle \qquad |\alpha|^2 + |\beta|^2 = 1$$

The most general single-qubit Hamiltonian involves the three Pauli matrices:

$$\hat{H} = -\frac{\hbar}{2}\left(\omega_0 \hat{\sigma}_z + \Omega\cos(\omega t)\hat{\sigma}_x\right)$$

where:
- **ω₀** is the qubit frequency (energy splitting between |0⟩ and |1⟩)
- **Ω** is the Rabi frequency (strength of the driving field)
- **ω** is the drive frequency

---

## Rabi Oscillations

When the drive is on resonance (ω = ω₀), the system undergoes **Rabi oscillations** — the population coherently oscillates between |0⟩ and |1⟩ at frequency Ω:

$$P_{|1\rangle}(t) = \sin^2\left(\frac{\Omega t}{2}\right)$$

This is the mechanism behind every quantum gate in a real quantum computer. A π-pulse (Ωt = π) flips |0⟩ → |1⟩ — that is a NOT gate. A π/2-pulse creates an equal superposition — that is a Hadamard gate.

Off resonance (ω ≠ ω₀), the oscillations are faster but incomplete — the population never fully transfers. The **generalized Rabi frequency** is:

$$\Omega_R = \sqrt{\Omega^2 + \Delta^2} \qquad \Delta = \omega - \omega_0$$

where Δ is the **detuning**.

---

## The Bloch Sphere

The qubit state maps to a point on the unit sphere:

$$|\psi\rangle \leftrightarrow \begin{pmatrix}\cos(\theta/2) \\ e^{i\phi}\sin(\theta/2)\end{pmatrix} \leftrightarrow (\sin\theta\cos\phi,\ \sin\theta\sin\phi,\ \cos\theta)$$

- **North pole** = |0⟩
- **South pole** = |1⟩  
- **Equator** = equal superpositions
- Time evolution = rotation of the Bloch vector

Rabi oscillations look like the Bloch vector spinning around an axis — one of the most intuitive visualizations in all of quantum mechanics.

## Connection to Your Existing App

This module is **architecturally separate** from the Schrödinger solver — it does not use the grid, Hamiltonian builder, or Crank-Nicolson stepper. The time evolution is a 2×2 matrix ODE, solved with:

```python
# backend: qubit_solver.py
from scipy.integrate import solve_ivp
import numpy as np

sigma_x = np.array([[0, 1], [1, 0]])
sigma_y = np.array([[0, -1j], [1j, 0]])
sigma_z = np.array([[1, 0], [0, -1]])

def rabi_rhs(t, psi, omega0, Omega, omega_drive):
    H = -0.5 * (omega0 * sigma_z + 
                Omega * np.cos(omega_drive * t) * sigma_x)
    return -1j * H @ psi

def solve_rabi(omega0, Omega, omega_drive, t_max, n_points):
    psi0 = np.array([1.0, 0.0], dtype=complex)  # start in |0⟩
    t_span = (0, t_max)
    t_eval = np.linspace(0, t_max, n_points)
    sol = solve_ivp(rabi_rhs, t_span, psi0, t_eval=t_eval,
                    args=(omega0, Omega, omega_drive),
                    method='RK45', rtol=1e-8)
    return sol.t, sol.y  # y shape: (2, n_points)
```

The Bloch sphere coordinates at each time step are:

```python
def bloch_vector(psi):
    # psi = [alpha, beta]
    x = 2 * np.real(psi[0] * np.conj(psi[1]))
    y = 2 * np.imag(psi[0] * np.conj(psi[1]))
    z = np.abs(psi[0])**2 - np.abs(psi[1])**2
    return x, y, z
```
